# Outcall Security Fix Summary — 2026-05-20

This summary condenses the private fix-wave notes into the public docs repo.
It is intended for beta testers who need to understand what changed since the
May 14 audit without reading the private root repository history.

## Release-blocking fixes completed

- The rule validator expands `$definition` references before CEL compilation.
- Shipped example rules and docs use the actual CEL bindings:
  `network.ip`, `network.port`, `dns.record_type`, and `agent.name`.
- CEL runtime failures are logged at `warn!` with rule id and file, then treated
  as no-match so default-deny still applies.
- The bridge forward chain uses `policy drop` with an explicit allow for
  unrelated non-Outcall traffic.
- DNS A/AAAA answers for private, loopback, link-local, ULA, multicast, and
  IPv4-mapped addresses are stripped unless the matching rule opts into
  `allow_private_ips`.
- `--no-proxy` refuses to start when loaded rules require `egress.mode: proxy`.
- Agent API session tokens require OS cryptographic randomness; there is no
  predictable fallback generator.

## Hardening added

- Dynamic direct-IP nftables rules are capped per container to limit DNS-driven
  rule-table growth.
- Agent API rate-limit state reaps stale container entries.
- Plain HTTP requests with mismatched absolute-form URI authority and `Host:`
  header are rejected.
- CONNECT rejects known non-HTTPS service ports by default while allowing
  rule-approved custom TLS ports.
- `.yaml` and `.yml` rule files are both loaded; empty rule sets emit warnings.

## Still deferred

- TLS interception is not implemented. HTTPS filtering is based on CONNECT host
  and SNI only.
- Release artifacts are not signed yet. SHA256 sums are produced; Sigstore/SBOM
  signing is planned for v0.2.
- A tighter `outcall-agent` seccomp profile is still planned.
