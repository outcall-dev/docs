# Outcall Security Audit — 2026-05-14

Scope: pre-public-release deep audit of the daemon, CLI, agent shim, and rule
engine. Audited against the project's load-bearing claim: **deny by default
unless a rule explicitly allows, enforced at the host so agents in the
container cannot bypass it.**

This audit builds on the May 5 internal pass (20 bypass vectors). Some of
those have been fixed; the rest are tracked here together with new findings.

| Severity | Count | Notes |
|---|---|---|
| Critical | 5 | Block release |
| High | 6 | Fix before public release |
| Medium | 7 | Schedule for next release |
| Low | 5 | Track but acceptable for v0.1 |
| Informational | 4 | Hardening recommendations |

---

## Critical

### C-0 — Rule validator skips `$definition` expansion, rejects valid rules

**File:** `application/outcalld/src/rules/engine.rs` (in
`validate_rule_file`).

The validator called from the agent rule-submit endpoint (`/v1/requests/rules`) compiles
`rule.condition` directly. It does not call `expand_definitions` first.
**Any rule that uses `$name` references — even valid ones — fails
validation with "Error parsing Invalid token at 0"** (CEL chokes on
`$`).

The full load path (`load_and_compile`) does expand, so rules loaded at
startup work; only the validator and agent-submitted rules are affected.

**Status: FIXED in this audit pass.** `validate_rule_file` now runs
`expand_definitions` before compiling. Regression test:
`outcalld/tests/example_rules_validation.rs` (the
`sentry-github-agent` example uses `$definitions` and would have failed
before the fix).

### C-1 — Rule file in repo doesn't actually work (broken default)

**File:** `rules.d/10-allow-dns-ports-ubuntu-com.yaml:12`

The shipped example uses `ip.dst == "..."` but the CEL evaluation context
exposes only `network`, `http`, `dns`, `docker`, `run`, `agent` (see
`engine.rs:393-398`). `ip.dst` references an undefined symbol, which raises a
CEL runtime error. That error is silently swallowed at `engine.rs:67`:

```rust
.unwrap_or(false);
```

The rule never matches. The agent is denied. The operator has no signal
beyond "Ubuntu apt mirror unreachable." This is fail-closed by accident, not
by design — and it makes the documented behaviour a lie.

**Why this is critical**: it proves the docs↔code drift is shipping
silently-broken rules. Any operator copy-pasting from `docs/rules.md` hits
this immediately.

**Fix**:
1. Rewrite `rules.d/10-allow-dns-ports-ubuntu-com.yaml` to use valid fields
   (`network.ip` instead of `ip.dst`).
2. Log CEL runtime errors at `warn!` level in `engine.rs::evaluate()` so
   broken rules surface in counters/logs.
3. Reconcile `docs/rules.md` with the actual CEL bindings (see C-2).

### C-2 — Documentation describes a different rule language than the daemon implements

**Files:** `docs/rules.md`, `application/outcall-api/src/lib.rs`,
`application/outcalld/src/rules/engine.rs`

`docs/rules.md` documents these CEL bindings that **do not exist** in the
shipped daemon:

| Doc says | Code actually exposes |
|---|---|
| `net.dst_ip`, `net.dst_port`, `net.protocol` | `network.ip`, `network.port`, `network.protocol` |
| `dns.type` | `dns.record_type` |
| `http.scheme` | not exposed |
| `agent.image` | not exposed (only `agent.name`) |
| `action: deny` | only `allow`, `block`, `enrich` (`outcall-api/src/lib.rs:70-76`) |
| `mode: intercept`, `match_body` | `EgressMode` enum has only `Proxy` and `DirectIp` |
| TLS interception, CA flags, `match_body` rules | not implemented |

Every operator following the docs writes a rule that either fails to load
(YAML schema error on `action: deny`, `mode: intercept`) or silently never
matches (`ip.dst`, `dns.type`).

**Why this is critical**: a security tool whose docs disagree with its
behaviour fails open in operator confusion — the operator ends up
sledgehammering a "block everything" rule that does work, or worse,
disabling the daemon. Either way, public release with this drift is
malpractice.

**Fix**: rewrite `docs/rules.md` to the actual implementation. If you want
the documented features, file them as future work. Don't ship docs that lie.
This audit ships with a corrected `docs/rules.md` and a `BINDING_REFERENCE`
section that's generated from the code (see follow-up task).

### C-3 — Bridge nftables `policy accept` rather than `policy drop` — **FIXED**

**Status: FIXED.** `application/outcalld/src/bridge.rs:150` now reads
`policy drop`. Non-outcall traffic is preserved by an explicit first rule
that accepts traffic on unrelated interfaces. The fix was verified in the
audit pass and is reflected in the release-blocking checklist below.

**Original finding (for audit trail):**

**File:** `application/outcalld/src/bridge.rs` (pre-fix)

```rust
type filter hook forward priority 0; policy accept;
iifname "{name}" ct state established,related accept
iifname "{name}" drop
oifname "{name}" ct state established,related accept
oifname "{name}" drop
```

The chain policy was `accept`. Isolation depended entirely on the two
`iifname/oifname … drop` rules continuing to be present and effective.

If anything goes wrong — a partial `nft -f` apply, a table flush from
another tool, an admin running `nft flush chain inet outcall forward` — the
chain falls open and **all forwarded traffic, including across our bridge,
is allowed**. There is no second line of defense.

**Why this is critical for a default-deny tool**: the deepest enforcement
layer should fail closed, not open. `policy drop` flips the failure mode:
if the explicit `accept` rules disappear, traffic is dropped instead of
allowed.

**Fix**: change the base ruleset to one of these patterns:

Option A — drop-by-default chain (recommended):
```
table inet outcall {
    chain forward {
        type filter hook forward priority 0; policy drop;
        iifname != "{name}" oifname != "{name}" accept
        iifname "{name}" ct state established,related accept
        oifname "{name}" ct state established,related accept
    }
}
```
First rule preserves unrelated host traffic (chain policy doesn't kick in
for it). Outcall-bridge traffic must be explicitly allowed by dynamic rules
or it dies in the policy default.

Option B — keep current policy but add a final-rule canary that screams if
the chain has been tampered with (logging-only, doesn't change semantics).

### C-4 — IPv6 forward path has no outcall-specific drop rules

**File:** `application/outcalld/src/bridge.rs:144-156`

The ruleset uses `inet` family, which **does** cover both IPv4 and IPv6 for
the `iifname/oifname` matches. So the base ruleset is fine.

**BUT:** dynamic allow rules inserted via `direct_ip` egress mode use
`ip saddr/daddr` and `ip6 saddr/daddr` (see `dns/mod.rs:298-328`). Need to
verify both paths are exercised — if the IPv6 insertion silently fails on a
host without ip6tables, agents resolving an AAAA record get an effective
deny while operator believes they got an allow. Check `dynamic/mod.rs` for
both-family insertion.

**Fix**:
- Add an integration test that asserts both IPv4 and IPv6 dynamic rules
  insert successfully (or that IPv6 insertion gracefully degrades with a
  visible warning).
- Document the requirement: "if you enable `direct_ip` egress for a
  host with AAAA records, the host kernel must have ip6_tables loaded."

## High

### H-1 — CEL runtime errors are silently swallowed

**File:** `application/outcalld/src/rules/engine.rs:55-71`

```rust
let matched = rule
    .program
    .execute(&cel_ctx)
    .ok()
    .and_then(|v| {
        if let Value::Bool(b) = v {
            Some(b)
        } else {
            None
        }
    })
    .unwrap_or(false);
```

A runtime error (undefined field reference, type mismatch, division by
zero) maps to "no match." The rule is skipped, evaluation continues. The
operator never sees a signal that their rule is broken.

This is fail-closed (good) but operationally hostile (bad). Combined with
C-1 it has already produced one silently-broken rule file in the repo.

**Fix:**
- Log at `warn!` (or `error!`) level when a CEL rule throws at runtime,
  including rule id, file, and the error.
- Add a per-rule "execution error" counter exposed via `outcall rules
  counters` so operators can spot ill-formed rules.
- Optionally: fail the whole rule set if any rule throws on its first
  evaluation against a synthetic context at load time (catch the
  obvious cases before they hit production traffic).

### H-2 — DNS rebinding detection only warns, does not block

**File:** `application/outcalld/src/dns/mod.rs:451-471`

```rust
if private {
    warn!(...);
}
```

When a public hostname resolves to RFC 1918 / loopback / link-local, we
log a warning and **forward the records to the client unchanged**. That's
the textbook DNS rebinding attack — an attacker-controlled DNS server can
make `evil.com` resolve to `127.0.0.1` or a container's gateway, letting
the agent talk to the host or other containers.

**Fix:**
- Filter private/loopback/link-local addresses out of upstream responses
  before returning them, unless an explicit allowlist names the host.
- Return NXDOMAIN if **all** answers were filtered (rather than an empty
  answer that some resolvers retry).
- Add an `outcall rules` flag `allow_private_resolution: true` if there's
  a legitimate use case (internal DNS pointing at private IPs).

### H-3 — `--no-proxy` mode disables most enforcement silently

**File:** `application/outcalld/src/main.rs` (daemon CLI), `proxy/mod.rs`

When the daemon runs with `--no-proxy`:
- HTTP and HTTPS rules become unenforceable (the only enforcement point is
  the L7 proxy).
- All `egress: { mode: proxy }` rules silently become useless — the proxy
  doesn't exist.
- Agents reach the destination via DNS + direct_ip or via raw socket if
  they bring their own resolver.

**Fix:**
- At startup, if `--no-proxy` is set AND any rule in the active rule set
  has `egress: { mode: proxy }`, refuse to start with a clear error.
- Otherwise, log a prominent `warn!` at startup naming exactly which layer
  is unenforced.

### H-4 — DirectIp egress can be flooded by an agent that controls DNS

**File:** `application/outcalld/src/dns/mod.rs:261-331`

For a rule with `egress: { mode: direct_ip }`, the daemon inserts one
nftables allow rule per (src_ip, dst_ip, port) tuple, for every IP returned
by upstream DNS.

A misconfigured (or malicious-via-DNS-server) agent can:
1. Issue thousands of A queries against an allowed hostname pointing at an
   attacker's DNS server.
2. The attacker's server returns a different IP each time.
3. The dynamic rule table grows unbounded.

**Fix:**
- Cap per-(container, rule) dynamic rule count (e.g., 256 entries).
- Add an LRU eviction strategy with a sane TTL (the dynamic rule manager
  already does some of this — verify the cap is enforced before insertion).
- Rate-limit `insert_rule` calls per container.

### H-5 — Rate-limit maps grow unbounded over container lifetime

**File:** `application/outcalld/src/agent_api/mod.rs:107-108`

```rust
perm_rate: Arc<Mutex<HashMap<String, SlidingWindow>>>,
rule_rate: Arc<Mutex<HashMap<String, SlidingWindow>>>,
```

Keyed by `container_id`. Containers get deleted, new ones get created. The
old entries are never reaped. Long-running daemons accumulate stale entries
indefinitely.

Also: every permission check holds the global mutex while doing the
sliding-window math. Under heavy concurrent agent traffic this is the
contention point.

**Fix:**
- Reap entries whose last timestamp is older than `2 * window`.
- Consider a sharded map or dashmap for less contention.
- Add a metric `agent_api_ratelimit_entries{type=perm|rule}` so operators
  notice unbounded growth.

### H-6 — `generate_token` falls back to a predictable RNG

**File:** `application/outcalld/src/agent_api/mod.rs:598-618`

```rust
fn fill_random(buf: &mut [u8]) {
    if let Ok(mut f) = std::fs::File::open("/dev/urandom") {
        ...
    } else {
        // Fallback (degraded but never blocks): mix time and pid.
```

If `/dev/urandom` is unavailable, we fall back to a deterministic LCG seeded
by time XOR pid. On Linux this should never happen, but if it does, **the
session tokens guarding all agent permission checks are guessable.**

**Fix:**
- Remove the fallback. Use `rand::rngs::OsRng` (already a transitive
  dependency) or `getrandom`. If the OS won't give us entropy, refuse to
  start — a security tool that fails to a guessable token is worse than
  one that won't run.

## Medium

### M-1 — `derive_agent_name` allows name collisions across replicas

**File:** `application/outcalld/src/agent_api/mod.rs:502-506`

`my-agent-1` and `my-agent-2` both resolve to `my-agent`. That's by
design (the replica suffix). But: `db-agent-2` and `db-agent` (no suffix)
both resolve to `db-agent` as well. If the operator names a container
`db-agent-12345`, it'll also be `db-agent`. A side-channel naming
collision lets one container claim another's identity.

**Fix:**
- At container creation/check-in time, enforce that the stripped agent
  name is unique among currently-running outcall-managed containers.
- Document the suffix-stripping rule prominently in `docs/rules.md`.

### M-2 — Proxy parses absolute-form URI host but ignores `Host:` header mismatch

**File:** `application/outcalld/src/proxy/mod.rs:391-454` (`handle_http`)

For plaintext HTTP, the proxy evaluates rules against the absolute-form
URI's host and forwards the request to that host. The `Host:` header is
forwarded verbatim. Some upstreams use the `Host:` header for routing —
the agent could bypass logging/observability by writing
`GET http://allowed.example.com/ HTTP/1.1\r\nHost: evil.com\r\n`.

This is a misuse of HTTP, but worth either normalizing or rejecting.

**Fix:**
- Reject requests where `Host` header ≠ URI authority (after lowercasing).
- Or: log a `warn!` and rewrite Host to match URI before forwarding.

### M-3 — CONNECT supports arbitrary ports

**File:** `application/outcalld/src/proxy/mod.rs:282-288`, `parse_host_port`

CONNECT to any host:port the rule engine allows. No restriction to
443/8443/etc. A loose rule like `http.host == "example.com"` allows
`CONNECT example.com:25` (SMTP), `CONNECT example.com:6667` (IRC),
`CONNECT example.com:11211` (memcached).

**Fix:**
- Add a default port allowlist (`[443]`) that rules must opt out of.
- Or: surface `network.port` in the rule context for CONNECT and
  require rules to match it explicitly. The current evaluation already
  passes `port` into `build_http_ctx` — the proxy just doesn't filter.

### M-4 — Agent shim has no seccomp profile

**File:** `application/outcall-agent/src/main.rs`, `application/outcalld/src/docker/mod.rs`

The agent shim runs in containers with `cap_drop ALL` but no seccomp
filter beyond Docker's default. A compromised agent process has the full
default syscall surface (~300 syscalls), including BPF-based loopback
tricks, abstract Unix sockets, and `unshare`.

**Fix:**
- Ship a `seccomp.json` profile with the daemon that allows only the
  syscalls outcall-agent needs.
- Apply it via `SecurityOpt: ["seccomp=/path/to/outcall-agent.json"]`
  when starting agent containers.

### M-5 — Rule reload is not atomic across the proxy/DNS/agent-API paths

**File:** `application/outcalld/src/rules/engine.rs:113-122`

`RuleEngine::reload()` swaps a single Arc. That's atomic for the rule
set — good. But the proxy, DNS handler, agent API, and dynamic rule
manager all read `Arc<RuleSet>` at different points in their request
handlers. A reload mid-request can still see a partial transition
(e.g., DNS evaluates against old set, proxy then evaluates against
new set for the same logical request).

In practice this is unlikely to be exploitable, but for an isolation
tool with a default-deny posture, "old set allowed it but new set
wouldn't" is the kind of edge case attackers chain.

**Fix:**
- Document the behavior explicitly: rules apply to *new* requests only;
  in-flight requests use whatever they bound at start.
- Or: add a generation counter to rule evaluations and refuse to
  forward a request whose rule-set generation differs across DNS and
  proxy checks.

### M-6 — No per-rule egress timeout

**File:** `application/outcalld/src/proxy/mod.rs:37`

`IDLE_TIMEOUT_SECS: u64 = 300` is a global. A rule that allows
long-running tunnels (websockets, gRPC streams) gets the same timeout
as one allowing API calls. Misconfigured tunnels eat resources and
delay graceful shutdown.

**Fix:** add optional `timeout_secs` to `EgressSpec`.

### M-7 — Empty rule files load as empty rule set instead of erroring

**File:** `application/outcalld/src/rules/engine.rs:235-241`

```rust
if yaml_files.is_empty() {
    info!(... "no .yaml files found — starting with empty rule set");
    return Ok(RuleSet::default());
}
```

`docs/rules.md:298` says: *"Empty rule files are rejected — `version: "1"`
with no `rules:` is a parse error."* But the code accepts an empty
directory and accepts a file with `version: "1"` and an empty `rules:` list.

An operator who mistakenly points `--rules-dir` at the wrong directory
gets a permissive default-deny (everything blocked, no rules) and might
not realize until traffic fails.

**Fix:** at startup, emit a prominent `warn!` if zero rules loaded, OR
require the operator to opt in with `--allow-empty-rules`. Match
documented behavior either way.

## Low

### L-1 — Rule files only matched on `.yaml`, not `.yml`

`engine.rs:230` — `e.path().extension().is_some_and(|ext| ext == "yaml")`.
`.yml` is common. Operators get a silently-missing rule file.

### L-2 — Health endpoint `/outcall-health` is not authenticated

`proxy/mod.rs:239` — anyone with proxy access gets back a 200/JSON. Low
risk because the proxy is bound to the daemon's local interface, but a
misconfigured `0.0.0.0` bind exposes a daemon fingerprint.

### L-3 — `block_until_done()` DNS server has no graceful drain

`dns/mod.rs:562` — shutdown is "tell select to stop." In-flight queries
are abandoned.

### L-4 — `info!()` logs include hostnames that may be sensitive

`dns/mod.rs:158-162`, `proxy/mod.rs:307`. Hostnames can leak business
context. Document that logs are sensitive; add a `--log-redact-hostnames`
flag for ops teams that need it.

### L-5 — `nft` is spawned as a subprocess rather than via netlink

`bridge.rs:160`. Shell-out is fine functionally but adds parsing
brittleness, depends on the `nft` binary being present and on PATH, and
the error path stringly-types failures (`BridgeError::Nftables(String)`).
Long-term: use the `nftnl-rs` or `mnl` crates.

## Informational / hardening

### I-1 — Add `cargo deny` to CI

Catches license issues, RUSTSEC advisories, dupe crates. `cargo audit` is
already enabled.

### I-2 — Ship a signed release

For a security-critical binary, GPG- or Sigstore-signed releases let
downstream operators verify provenance. Tap `Outcall-dev/homebrew-tap`
should pin to sha256 of signed artifacts.

### I-3 — Drop unused workspace deps

`cargo machete` shows several unused deps in `outcall` crate. Each one is
a future supply-chain risk.

### I-4 — Document the trust boundary in `SECURITY.md`

Public projects need a vuln disclosure address. See `docs/security/SECURITY.md`
shipped with this audit.

## What's already solid

It's worth being explicit about what works:

- **Host-level enforcement is real.** `scripts/e2e/tests/14-security-boundary.sh`
  proves an agent inside the container cannot bypass nftables by flushing
  iptables, modifying its own routing, or trying to add nft rules in the
  container namespace. The boundary is correct.
- **SO_PEERCRED-based identity is correctly implemented.** Agents cannot
  claim to be a different container — the host derives identity from the
  Unix socket peer credentials, not from any agent-provided field.
- **Default-deny in the rule engine itself is correct.** No-match returns
  `Decision::Block`. Timeout in the agent API returns `allowed: false`.
- **The proxy enforces pre-CONNECT-200 and post-SNI evaluation** so a
  client sending an unrelated SNI after CONNECT gets the connection torn
  down (rather than the CONNECT host being authoritative forever).
- **TLS is not terminated.** The proxy peeks ClientHello bytes only. There
  is no CA infrastructure to compromise.

## Release-blocking checklist

Before public release, these must be fixed (Critical + the High items that
shape the threat model):

- [x] C-0: validator now expands `$definitions` before CEL compile
- [x] C-1: shipped `rules.d/10-allow-dns-ports-ubuntu-com.yaml` rewritten to valid CEL fields
- [x] C-2: `docs/rules.md` reconciled with actual code (`network.*` not `net.dst_*`, `dns.record_type` not `dns.type`, `agent.image` removed, `action: deny` and `mode: intercept` removed)
- [x] C-3: bridge switched to `policy drop` with positive allow for non-outcall traffic
- [ ] C-4: verify or document IPv6 dynamic rule insertion (still open)
- [x] H-1: CEL runtime errors now surface at `warn!` with rule_id and file
- [ ] H-2: block DNS rebinding, don't just warn
- [ ] H-3: refuse to start (or warn loudly) when `--no-proxy` + proxy rules
- [ ] H-6: remove predictable RNG fallback for session tokens

Items not on this list are tracked but acceptable for a clearly-versioned
v0.1 release with a `SECURITY.md` that names the residual risks.
